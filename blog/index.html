01 Why LoRA Matters for Multi-Tenant LLM Serving

Running a single large language model for a single purpose is, relatively speaking, a solved problem. The real challenge begins when you need to serve dozens or hundreds of specialized variants of that model to different customers, all from the same GPU fleet. This is the reality of multi-tenant LLM platforms, and it is where LoRA becomes not just useful but essential.

What LoRA Actually Does

Low-Rank Adaptation (LoRA) is a parameter-efficient fine-tuning method introduced by Hu et al. (2021). Instead of updating all the billions of parameters in a large language model during fine-tuning, LoRA freezes the pre-trained weights and injects small, trainable rank-decomposition matrices into each transformer layer. During inference, these matrices are applied as lightweight modifications to the base model's behavior.

The key insight is dimensional: a weight matrix in a 70B-parameter model might have dimensions of 8192 × 8192 (67 million parameters per layer). A LoRA adapter with rank 16 replaces this with two matrices of dimensions 8192 × 16 and 16 × 8192 — just 262,144 parameters. That is a 99.6% reduction in trainable parameters per layer.

Key Insight

A LoRA adapter for a 70B-parameter model is typically 10–100 MB in size, compared to 140 GB for the full model weights. That is 0.01–0.07% of the base model — small enough to store hundreds of adapters on a single node.

The Multi-Tenant Economics

Consider a platform serving 50 enterprise customers, each requiring their own fine-tuned model. Without LoRA, you have two unpleasant options:

  • 50 full model copies: At 140 GB per model (for a 70B at FP16), that is 7 TB of GPU memory just for weights. Even with the largest available GPUs, you are looking at dozens of nodes dedicated purely to storing redundant copies of nearly identical models.
  • One model, no customization: You sacrifice the specialization that customers are paying for.

LoRA provides a third path: one base model plus 50 adapters. The base model occupies its 140 GB once. Each adapter adds 10–100 MB. Even at the upper bound, all 50 adapters together consume only 5 GB — less than 4% of the base model. The economics shift from "linear cost scaling per customer" to "near-zero marginal cost per customer."

Approach GPU Memory Cost / Customer Customization
Full fine-tuned copies 140 GB × N High (linear) Full
Shared base, no adapters 140 GB × 1 Fixed None
Base + LoRA adapters ~145 GB total Near-zero marginal Full

Serving engines like vLLM have made multi-LoRA serving practical by supporting hot-loading of adapters and batching requests across different LoRA variants within the same inference batch. A single vLLM instance can serve multiple adapters simultaneously, dynamically loading and unloading them as demand shifts.

But the ability to serve multiple adapters per pod introduces a new problem: which pod should each request go to? When the routing layer does not understand adapter state, the answer is usually the wrong pod.

02 The Adapter Swap Problem at Scale

When a request arrives at a vLLM pod requesting LoRA adapter legal-qa-v3, but that pod currently has medical-summary-v2 loaded, the serving engine must perform an adapter swap: unload the current adapter's weights from GPU memory and load the requested adapter from CPU memory or storage.

The Cost of a Single Swap

An adapter swap involves several steps, each with measurable latency:

  1. Deallocate the current adapter's weight matrices from GPU HBM — typically near-instant but can trigger memory compaction.
  2. Transfer the new adapter's weights from host memory (or NVMe, or network storage) to GPU HBM. For a 50 MB adapter over PCIe Gen4 x16, this takes approximately 1–3 ms. For a 500 MB adapter, 10–30 ms.
  3. Initialize the adapter in the serving engine's computation graph, including any necessary reshaping or memory layout adjustments.

End to end, a typical adapter swap on modern hardware takes 50–200 ms, depending on adapter size, storage tier (GPU memory → CPU memory → NVMe), and whether the engine needs to drain its current batch before swapping.

The batch-draining penalty

Many serving engines cannot swap adapters mid-batch. If a pod is processing a batch of 32 requests for adapter A and receives a request for adapter B, it must either finish the entire batch before swapping (adding batch-completion latency) or hold the request in queue until the swap completes. Either way, the requesting user pays a delay far exceeding the raw swap time.

When Random Routing Meets Real Traffic

Consider a cluster of 8 vLLM pods serving 16 different LoRA adapters. With round-robin load balancing — the default for most Kubernetes ingress controllers — each pod receives a roughly uniform mix of requests across all 16 adapters.

If each pod can hold 2–4 adapters in GPU memory at once, a round-robin distribution means every pod swaps adapters on 75–87% of incoming requests. At 100 requests per second per pod, that translates to 75–87 swap events per second per pod.

The CPU Context-Switching Analogy

Operating systems researchers discovered decades ago that naive process scheduling creates convoy effects — when the scheduler rotates through processes too aggressively, the system spends more time switching between processes than actually executing them. The same principle applies to adapter swaps.

Each individual swap might cost only 100 ms. But 80 swaps per second per pod means 8 full seconds of swap overhead per second — obviously impossible, which means requests queue up, latencies spike, and throughput collapses. In practice, the system degrades to the point where GPU utilization plummets despite a full request queue, because the GPUs spend most of their time waiting for adapter weights to transfer.

The solution is the same one that fixed CPU scheduling: affinity-aware routing. Route requests to the pod that already has the right adapter loaded, and swap only when genuinely necessary. This is exactly what llm-d's Endpoint Picker does.

03 How llm-d's EPP Scores Adapter Affinity

The Endpoint Picker (EPP) is llm-d's intelligent routing layer, implemented as an ext-proc filter in the Envoy-based Gateway API data path. When a request arrives, the EPP evaluates every available pod (endpoint) and assigns a composite score to determine the optimal routing target. In llm-d v0.5, adapter affinity was added as a first-class scoring signal.

The Multi-Factor Scoring Pipeline

The EPP does not route based on a single heuristic. Instead, it runs each candidate pod through a filter/scorer pipeline that evaluates multiple dimensions and produces a weighted composite score. The key scoring factors, in approximate order of weight:

Adapter Affinity
1.0
KV Cache Match
0.6
Queue Depth
0.3
GPU Utilization
0.2

The relative weights are configurable, but the default hierarchy reflects a clear insight: avoiding an adapter swap is almost always more valuable than any other single optimization. The cost of swapping an adapter (50–200 ms of latency plus potential batch drain) dwarfs the cost difference between routing to a slightly more or less loaded pod.

How Adapter Affinity Scoring Works

Each vLLM pod periodically reports its state to the EPP, including which LoRA adapters are currently loaded in GPU memory. When a request arrives with an adapter specified (via the model field in the OpenAI-compatible API, which maps to a LoRA adapter name in vLLM), the EPP evaluates each pod:

  • Adapter loaded and active: Full affinity score. The pod can serve the request immediately with zero swap overhead.
  • Adapter loaded but inactive: High affinity score. The adapter is in GPU memory but not currently being used in the active batch — activation is near-instant.
  • Adapter not loaded, capacity available: Partial affinity score. The pod has free adapter slots and can load the adapter without evicting another — a swap is needed but with no cascading eviction.
  • Adapter not loaded, at capacity: Low affinity score. The pod must evict an existing adapter to make room, which could disrupt other request flows.
conceptual// Simplified view of the EPP scoring logic for adapter affinity
func scoreAdapterAffinity(pod PodState, requestedAdapter string) float64 {
    if pod.HasActiveAdapter(requestedAdapter) {
        return 1.0  // Best case: already loaded and serving
    }
    if pod.HasCachedAdapter(requestedAdapter) {
        return 0.8  // Loaded in memory, needs activation
    }
    if pod.HasAdapterCapacity() {
        return 0.3  // Can load without evicting
    }
    return 0.0      // Must evict to load
}

Composite Scoring in Practice

The adapter affinity score is combined with the other factors to produce a final routing decision. Conceptually: 

conceptualscore(pod) = w_adapter × adapterAffinity(pod)
           + w_kvcache × kvCacheMatch(pod, request.prompt)
           + w_queue   × (1 - normalize(pod.queueDepth))
           + w_gpu     × (1 - normalize(pod.gpuUtilization))

The EPP selects the pod with the highest composite score. When multiple pods tie (for example, three pods all have the requested adapter loaded), the tie is broken by the secondary factors — KV cache hit rate, then queue depth. This means that among pods with the right adapter, traffic naturally flows to the one with the best cache locality and lowest load.

The Emergent Clustering Effect

An elegant property of affinity-aware routing is that it creates natural adapter clustering without explicit assignment. There is no static mapping of "adapter X always goes to pod Y." Instead, the first request for a given adapter goes to the least-loaded pod (since no pod has affinity). Subsequent requests for that adapter are routed to the same pod because it now has the highest affinity score. Over time, each pod accumulates a working set of 2–4 frequently-used adapters, and requests cluster accordingly.

This is self-healing, too. If a pod restarts and loses its adapter state, the EPP temporarily routes that adapter's traffic to other pods that still have it loaded. As the restarted pod begins receiving requests, it rebuilds its adapter working set organically.

04 Real-World Use Cases

Cache-aware LoRA routing is not an abstract optimization. It unlocks specific architectural patterns that are impractical without intelligent adapter placement. Here are several production scenarios where this capability makes a material difference.

Multi-Customer SaaS Platforms

The most direct application is per-customer model customization on a shared infrastructure platform. A SaaS company offering AI-powered document processing might serve 200 enterprise customers, each with distinct requirements:

  • Tone and style: Customer A wants formal, third-person outputs; Customer B wants concise, action-oriented bullet points.
  • Domain vocabulary: A pharmaceutical company needs the model trained on its specific drug names, compound identifiers, and internal terminology.
  • Compliance patterns: A financial services customer requires outputs that conform to specific regulatory disclosure formats.
  • Language mix: Some customers operate primarily in English, others in German or Japanese, and fine-tuning on their specific bilingual patterns improves quality.

Each customer gets a LoRA adapter trained on their data and preferences. With cache-aware routing, the platform can serve all 200 customers from a modest GPU fleet, with each pod naturally specializing in a cluster of related customers. The routing is entirely transparent — customers send requests to a single API endpoint, and the EPP handles placement.

Domain-Specific Assistant Suites

Organizations increasingly deploy multiple specialized AI assistants rather than one general-purpose chatbot. A hospital system might run:

  • clinical-notes-v2 — generates structured clinical documentation from physician dictation
  • patient-qa-v1 — answers patient questions in accessible, non-technical language
  • radiology-report-v3 — produces standardized radiology reports from findings
  • billing-code-v1 — suggests ICD-10 and CPT codes from clinical notes

All four assistants share the same base model (for example, Llama 3.1 70B) with different LoRA adapters. With cache-aware routing, the EPP clusters clinical-notes and radiology-report traffic to the same pods (both are physician-facing, high-volume during morning rounds), while patient-qa traffic routes to pods that are less loaded during clinical hours but busier during evening patient portal usage.

A/B Testing and Canary Deployments

LoRA adapters make model experimentation operationally lightweight. To A/B test a new fine-tuning approach:

  1. Deploy the new adapter alongside the existing one: legal-qa-v3 (current) and legal-qa-v4-candidate (new).
  2. Configure the request gateway to label 10% of traffic with the candidate adapter name.
  3. The EPP naturally routes candidate traffic to one or two pods that load the new adapter, while the other pods continue serving the current version undisturbed.
  4. Compare quality metrics. If the candidate wins, gradually shift traffic. If it loses, remove the adapter — no pods were disrupted beyond the test group.

Without cache-aware routing, a 10% canary deployment would contaminate every pod with swap overhead for the candidate adapter, degrading performance for the 90% of traffic on the stable version. With the EPP's affinity scoring, the blast radius is contained.

Regional and Regulatory Adapters

Global deployments often need region-specific model behavior. A content moderation system might use different adapters for different regulatory environments:

  • moderation-eu-v2 — tuned for EU Digital Services Act requirements, stricter on certain content categories
  • moderation-us-v2 — tuned for US content policies with different thresholds
  • moderation-apac-v1 — handles CJK-language content with specialized tokenization awareness

Traffic from each region naturally clusters by adapter, and the EPP ensures that EU traffic consistently routes to pods with the EU adapter loaded. This is particularly valuable when combined with data residency requirements — pods in EU regions can be pre-warmed with the EU adapter, and the EPP's affinity scoring keeps traffic pinned there without hard-coding routing rules.

05 Cache-Aware LoRA Routing with Disaggregated Prefill/Decode

llm-d's architecture supports prefill/decode (P/D) disaggregation — a technique that separates the two phases of LLM inference onto different GPU pools. This section explores how LoRA routing interacts with disaggregated inference and why the combination is particularly powerful.

Understanding Prefill/Decode Disaggregation

LLM inference has two distinct computational phases with very different resource profiles:

  • Prefill processes the entire input prompt in parallel. It is compute-bound (high FLOPS utilization, high arithmetic intensity) and runs once per request. For a 4,096-token prompt on a 70B model, prefill might take 500–1,500 ms.
  • Decode generates output tokens one at a time (or a few at a time with speculative decoding). It is memory-bandwidth-bound (low arithmetic intensity, dominated by weight reads) and runs for every output token.

When both phases run on the same GPU, they compete for resources inefficiently. Prefill wants high compute utilization; decode wants high memory bandwidth. Disaggregation assigns each phase to GPU pools optimized for its workload profile, connected by a KV cache transfer mechanism.

The LoRA Routing Challenge in P/D

Disaggregation adds a layer of complexity to LoRA routing because both the prefill and decode pods need the correct adapter loaded:

  • The prefill pod needs the adapter to correctly compute the key-value cache entries for the input prompt. LoRA modifies the attention projections, so the KV cache produced by prefill must reflect the adapter's weight modifications.
  • The decode pod needs the same adapter to generate output tokens that are consistent with the adapted model's behavior. It also needs the adapter to correctly compute each new KV cache entry during autoregressive generation.

If the routing layer sends a request to a prefill pod with the right adapter but a decode pod without it, the decode pod must swap adapters mid-stream — paying the swap penalty at the worst possible time, when the user is waiting for token-by-token output.

How llm-d Handles This

The EPP's routing pipeline is P/D-aware. When disaggregated serving is enabled, the routing decision considers adapter affinity at both stages:

  1. Prefill routing: The EPP selects a prefill pod using the standard composite score, with adapter affinity as the dominant factor. This ensures the KV cache is computed with the correct adapter applied.
  2. KV cache transfer: After prefill completes, the computed KV cache is transferred to a decode pod. The EPP coordinates this handoff, selecting a decode pod that already has the same adapter loaded.
  3. Decode routing: The selected decode pod has both the KV cache (received from the prefill pod) and the correct adapter. Generation proceeds without any swap overhead.

The adapter affinity signal propagates through the entire pipeline. This means the EPP effectively maintains two adapter affinity maps — one for the prefill pool and one for the decode pool — and ensures both phases of a request land on pods with the correct adapter.

Performance Impact

The combination of P/D disaggregation with cache-aware LoRA routing yields compounding benefits. Disaggregation alone can improve throughput by 30–60% for mixed workloads. Adding adapter-aware routing eliminates the swap overhead that would otherwise erode those gains in multi-adapter scenarios. Together, they enable higher GPU utilization, lower tail latency (P99), and more predictable per-request performance even as the number of active adapters grows.

Why This Combination Matters at Scale

Without P/D disaggregation, a large multi-adapter deployment faces a tension: you want pods to specialize on a small set of adapters (for affinity), but you also want to balance compute load evenly (for utilization). A pod that has become the "legal-qa specialist" might be overwhelmed during a burst of legal traffic while other pods sit idle.

Disaggregation relaxes this tension. Prefill pods can be more aggressively specialized by adapter (since prefill is a short, bursty operation), while decode pods can be more flexibly scheduled (since decode is a longer, bandwidth-bound operation that benefits from batching across multiple concurrent requests). The EPP manages the balance automatically, using adapter affinity as the primary signal at both stages while allowing load balancing to act as the tiebreaker.
