01 LoRA Mechanics Deep Dive

Before we can understand why adapter-aware routing matters, we need to understand exactly what a LoRA adapter is and how it modifies a running model. Low-Rank Adaptation (LoRA), introduced by Hu et al. (2021), is a parameter-efficient fine-tuning technique that has become the dominant method for customizing large language models in production. Its elegance lies in a simple mathematical observation about weight updates during fine-tuning.

The Low-Rank Decomposition

During standard fine-tuning, every weight matrix W in the model is updated by a delta matrix ΔW, producing a new effective weight W' = W + ΔW. Research has shown that these delta matrices tend to have very low intrinsic rank — meaning the information in the update can be captured by a much smaller pair of matrices. LoRA exploits this by decomposing the update into two low-rank matrices: 

mathW' = W + B × A

where:
  W  is the original frozen weight matrix  (d × k)
  B  is the down-projection matrix          (d × r)
  A  is the up-projection matrix            (r × k)
  r  is the rank (typically 4 to 64)        r << min(d, k)

The original weight matrix W is never modified. During inference, the adapter's contribution is computed as an additive overlay: for an input x, the output becomes Wx + BAx. The matrices B and A are the only trainable parameters during fine-tuning, and they are the only parameters stored and transferred when loading an adapter.

Why Rank Choice Matters

The rank r is the single most important hyperparameter in a LoRA adapter. It controls the tradeoff between adapter expressiveness and size:

  • Rank 4: Extremely compact. For a 70B model with 8192-dimensional hidden states, each adapted weight matrix adds only 2 × 8192 × 4 = 65,536 parameters. Total adapter size across all attention layers comes in around 12–15 MB. Suitable for lightweight style or tone adjustments where the base model already has strong domain knowledge.
  • Rank 16: The most common production choice. Each weight matrix adds 262,144 parameters per adapted layer. Total adapter size for a 70B model is approximately 50 MB. This rank captures meaningful domain specialization while remaining small enough to swap quickly and store many adapters simultaneously.
  • Rank 64: High capacity. Each weight matrix adds over 1 million parameters, bringing total adapter size to approximately 200 MB. Used when significant behavioral changes are needed — for example, training a model to follow complex output schemas or to handle an entirely new domain like code generation in a niche programming language.
  • Rank 128+: Rarely used in production. At this point, the adapter approaches the size where full fine-tuning of select layers may be more efficient. Adapter size exceeds 400 MB, and the swap cost begins to erode the latency benefits of the LoRA approach.
Key Insight

Adapter size directly impacts routing decisions. A rank-16 adapter at ~50 MB can be transferred to GPU HBM in under 5 ms over PCIe Gen5. A rank-64 adapter at ~200 MB takes 15–20 ms. When you are making routing decisions at the microsecond level, this 4x difference in swap cost is significant enough to change which pod is optimal.

How Adapters Modify Attention

LoRA adapters are typically applied to the attention projection matrices in each transformer layer. A standard multi-head attention block has four weight matrices:

  • Q (Query) projection: Maps the input to query vectors. LoRA on Q modifies what the model "looks for" in the input.
  • K (Key) projection: Maps the input to key vectors. LoRA on K modifies how the model represents what information each position "offers."
  • V (Value) projection: Maps the input to value vectors. LoRA on V modifies the actual content that gets aggregated during attention.
  • O (Output) projection: Maps concatenated attention heads back to the model's hidden dimension. LoRA on O modifies how attention results are combined.

Most production adapters target Q and V projections, following the original LoRA paper's finding that these provide the best quality-per-parameter tradeoff. Some adapters also include K and O projections for higher-quality adaptation, roughly doubling the adapter size. The choice of which projections to adapt is fixed at training time and becomes part of the adapter's metadata that the serving engine needs to know when loading it. 

architectureTransformer Layer (with LoRA adapters on Q, V)
================================================

Input x (hidden_dim=8192)
    |
    +---> Q = (W_q + B_q * A_q) * x     # adapted query
    +---> K =  W_k * x                   # original keys
    +---> V = (W_v + B_v * A_v) * x     # adapted values
    |
    +---> Attention(Q, K, V)

    |

Output

This architecture means that the KV cache produced during prefill is adapter-specific. The key and value tensors stored in the cache reflect the adapted weight matrices. If a different adapter is used during the decode phase, the KV cache entries will be inconsistent with the model's expectations, producing degraded or nonsensical output. This is why both the prefill and decode pods in a disaggregated setup must have the same adapter loaded — a point we will return to in Section 8.

02 The Adapter Swap Cost Model

Understanding the true cost of an adapter swap requires looking beyond the simple data-transfer calculation. A swap is not merely "copy 50 MB to GPU memory." It is a multi-stage process with both direct latency costs and indirect throughput penalties that compound under load.

GPU Memory Management

GPU High Bandwidth Memory (HBM) is a flat address space managed by the CUDA runtime and the serving engine's memory allocator. In a running vLLM instance, HBM is divided among several consumers:

  • Base model weights: The largest consumer. A 70B model in FP16 occupies approximately 140 GB. With INT8 quantization, this drops to roughly 70 GB; with INT4 (GPTQ/AWQ), approximately 35 GB.
  • KV cache pool: Pre-allocated memory for storing key and value tensors across active sequences. Typically sized at 20–40% of remaining memory after model weights.
  • Adapter weight slots: Memory reserved for one or more LoRA adapter weight sets. Each slot must be large enough for the largest adapter the pod might serve.
  • CUDA workspace: Scratch memory for intermediate computations, activation tensors, and NCCL communication buffers in multi-GPU setups.

When a swap occurs, the serving engine must deallocate the outgoing adapter's memory and allocate space for the incoming adapter. If the adapters are different sizes (a rank-16 adapter being replaced by a rank-64 adapter), the allocator may need to coalesce free blocks or compact memory to create a contiguous region large enough. This fragmentation overhead is often overlooked but can add 2–10 ms to the swap, particularly on GPUs that are running near their memory capacity.

The Batch-Draining Penalty

The batch drain is often the largest single contributor to swap latency. Most serving engines cannot atomically swap adapter weights while a batch is in-flight because the adapter weights are referenced by active CUDA kernels. The engine must wait for all sequences in the current batch to either complete their current forward pass or reach a safe synchronization point. At high batch sizes (32–64 sequences), this drain can take 200–500 ms for long-context sequences, dwarfing all other swap costs combined.

The Convoy Effect

The convoy effect, a term borrowed from operating systems scheduling theory, describes what happens when adapter swaps cascade through a serving cluster. The mechanism is straightforward but devastating:

  1. Pod A receives a request for adapter X, but currently has adapter Y loaded. It begins draining its batch to swap.
  2. While Pod A is draining, it cannot accept new requests. The load balancer redirects Pod A's share of traffic to Pods B and C.
  3. Pods B and C are now overloaded. Their queue depths spike, causing the EPP (or any load-aware balancer) to send traffic elsewhere.
  4. The redirected traffic includes requests for adapters not loaded on the receiving pods, triggering additional swaps.
  5. Multiple pods are now simultaneously draining batches and swapping adapters. Cluster-wide throughput collapses even though individual GPU utilization metrics still look healthy (the GPUs are "busy" doing memory management, not inference).

In extreme cases, this convoy effect can reduce effective cluster throughput to 10–20% of theoretical capacity. The cluster has plenty of GPU compute available, but the compute is gated behind a wall of serialized adapter swaps. This is the fundamental problem that cache-aware routing exists to solve.

03 EPP's Scoring Function for Adapter Affinity

The Endpoint Picker (EPP) is llm-d's intelligent routing layer, implemented as an ext-proc filter in the Envoy-based Gateway API data path. When a request arrives, the EPP evaluates every available pod and assigns a composite score to determine the optimal routing target. Understanding how this scoring function works — and why the weights are set the way they are — is central to understanding llm-d's approach to multi-adapter serving.

The Filter/Scorer Pipeline

The EPP processes candidate pods in two stages. First, a filter stage eliminates pods that cannot serve the request at all — for example, pods that are in a draining state, pods that have exceeded their maximum queue depth, or pods that do not support the requested model type. Second, a scorer stage evaluates the remaining candidates across multiple dimensions and produces a composite score.

This two-stage design is critical for performance. The filter stage is cheap (simple boolean checks) and typically eliminates 30–50% of candidates, reducing the number of expensive scoring evaluations. The scorer stage then runs the full multi-signal evaluation only on viable pods.

The Scoring Formula

The EPP's composite score for each candidate pod is a weighted sum of four normalized signals: 

scoring functionscore = 0.50 * adapter_affinity
     + 0.20 * kv_cache_affinity
     + 0.15 * (1 - queue_depth / max_queue_depth)
     + 0.15 * (1 - gpu_utilization / 100)

Each signal is normalized to the [0, 1] range before weighting, ensuring that the weights directly represent relative importance.

Why Adapter Affinity Gets 50% Weight

The weight distribution is not arbitrary. It reflects a cost asymmetry argument: the cost of routing a request to a pod without the correct adapter is categorically different from the cost of routing to a slightly more loaded pod.

Routing to a pod with queue depth 10 instead of queue depth 5 adds perhaps 50–100 ms of queueing delay. Routing to a pod without the correct adapter adds 85–670 ms of swap latency, plus the indirect cost of batch draining that delays every other request on that pod. The swap penalty is 2–10x larger than the load-balancing penalty in the common case, and can be 50x larger in the worst case. A 50% weight on adapter affinity is, if anything, conservative.

How Pod State Reporting Works

The EPP's scoring accuracy depends on timely information about each pod's state. In llm-d, vLLM pods report their state to the EPP through a lightweight health endpoint that provides:

  • Loaded adapters: The list of LoRA adapter names currently resident in GPU memory, along with their activity status (active in current batch, cached but idle, or in the process of loading).
  • Adapter capacity: How many additional adapter slots are available, based on the pod's memory budget and current allocation.
  • Queue depth: The number of requests currently waiting or in-flight on the pod.
  • GPU utilization: Current compute utilization percentage, aggregated across all GPUs in a multi-GPU pod.
  • KV cache state: Prefix hash information for cache-aware prompt routing.

This state is reported at a configurable interval, typically every 500 ms to 2 seconds. The EPP maintains a local cache of pod states and uses the most recent report for scoring. A staleness window is applied — if a pod has not reported in more than 5 seconds, it is filtered out of the candidate set.

Adapter Affinity Scoring in Detail

The adapter affinity signal is not a simple binary match. The EPP distinguishes between multiple levels of affinity: 

pseudocodefunc scoreAdapterAffinity(pod PodState, requestedAdapter string) float64 {
    if pod.HasActiveAdapter(requestedAdapter) {
        return 1.0    // Best: loaded and actively serving this adapter
    }
    if pod.HasCachedAdapter(requestedAdapter) {
        return 0.8    // Good: in GPU memory, needs kernel activation
    }
    if pod.HasAdapterInCPU(requestedAdapter) {
        return 0.4    // Moderate: in host memory, needs PCIe transfer
    }
    if pod.HasAdapterCapacity() {
        return 0.3    // Acceptable: can load from storage without eviction
    }
    return 0.0        // Worst: must evict another adapter to make room
}

Tie-Breaking Logic

When multiple pods have the same composite score — the most common scenario being two pods that both have the requested adapter actively loaded — the EPP applies a deterministic tie-breaking cascade:

  1. KV cache affinity: If the request includes a prefix that matches cached KV entries on one pod, that pod wins the tie. This is common for multi-turn conversations where the prompt prefix matches the previous turn.
  2. Queue depth: Among remaining ties, the pod with the shortest queue wins. This provides natural load balancing within an adapter's affinity group.
  3. Random jitter: If ties persist (rare), a small random perturbation prevents pathological herding where all requests for a given adapter pile onto the same pod.

The Emergent Clustering Effect

One of the most elegant properties of affinity-aware routing is that it produces natural adapter clustering without any static configuration. There is no mapping table that says "adapter X always goes to pod Y." Instead, clustering emerges dynamically from the scoring function:

  1. The first request for a new adapter goes to the least-loaded pod with available adapter capacity (since no pod has affinity for an unseen adapter, load-based signals dominate).
  2. That pod loads the adapter and now has the highest affinity score for it.
  3. Subsequent requests for the same adapter route to the same pod, reinforcing its specialization.
  4. Over time, each pod accumulates a working set of 2–6 frequently-requested adapters.

This emergent clustering is self-healing. If a pod restarts and loses its adapter state, the EPP temporarily routes that adapter's traffic to other pods that still have it loaded. As the restarted pod begins receiving requests, it rebuilds its adapter working set organically. No operator intervention is required.

04 Multi-Adapter Serving Architecture

A common misconception is that each GPU pod can only hold one LoRA adapter at a time, making swaps an inevitable consequence of serving multiple adapters. In reality, modern serving engines support simultaneous multi-adapter serving, and understanding this capability is essential for designing an effective routing strategy.

Shared Base, Additive Overlays

The base model weights are loaded once into GPU HBM and shared across all requests, regardless of which adapter they target. LoRA adapters are stored separately as small additive overlays. During a forward pass, the serving engine applies the appropriate adapter by adding the low-rank product BA to the base weight for each adapted layer. This means that multiple adapters can coexist in GPU memory simultaneously — they occupy separate memory regions and do not interfere with each other.

| CUDA Workspace               |  ~5 GB         |

Batching Across Adapters

vLLM's multi-LoRA support enables something powerful: batching requests that target different adapters in the same inference batch. The base model's forward pass is computed once for the entire batch. Then, for each adapted layer, the engine applies the correct adapter's BA matrices to each sequence's activations based on which adapter that sequence requires.

This is possible because the LoRA computation is a simple matrix multiply-add operation that can be parallelized across sequences. The engine maintains a mapping from each sequence in the batch to its adapter, and the adapter application step uses this mapping to gather the correct weights. The overhead of multi-adapter batching is small — typically less than 5% compared to single-adapter batching — because the adapter computation is dominated by the base model's much larger weight matrices.

Memory Budget Calculation

The number of adapters a pod can hold simultaneously is determined by a simple memory budget equation: 

memory budgetavailable_for_adapters = total_hbm
                      - base_model_size
                      - kv_cache_pool_size
                      - cuda_workspace

adapter_slots = floor(available_for_adapters / max_adapter_size)

Example (H100 80GB, INT4 70B model):
  available = 80 - 35 - 30 - 5 = 10 GB
  rank-16 adapters: 10 GB / 50 MB  = ~200 slots
  rank-64 adapters: 10 GB / 200 MB = ~50 slots

In practice, the number of simultaneously active adapters is constrained more by the serving engine's management overhead than by raw memory capacity. vLLM currently supports up to several hundred registered adapters per instance, with a configurable limit on simultaneously loaded adapters that defaults to values in the range of 8–64, depending on adapter size and memory pressure.

Eviction Policies

When a pod's adapter slots are full and a request arrives for a new adapter, the serving engine must evict one adapter to make room. The eviction policy determines which adapter is removed:

  • LRU (Least Recently Used): The adapter that has gone the longest without serving a request is evicted. This is the default policy in most serving engines and works well when adapter popularity follows a long-tail distribution.
  • Frequency-based: Track request counts per adapter over a sliding window. Evict the adapter with the lowest request frequency. This policy is more resistant to one-off requests polluting the cache.
  • Size-aware LRU: Factor adapter size into the eviction decision. When choosing between evicting a 50 MB rank-16 adapter and a 200 MB rank-64 adapter that was used slightly more recently, it may be worth evicting the larger adapter if doing so frees enough space for the incoming adapter plus an additional slot.
Key Insight

The EPP's adapter affinity scoring and the pod-level eviction policy work together as a two-tier optimization. The EPP minimizes evictions by routing requests to pods that already have the adapter. When evictions do occur (because no pod has the adapter, or affinity pods are overloaded), the pod-level eviction policy ensures the least-disruptive adapter is removed. Neither layer alone is sufficient — routing without smart eviction leads to random churn, and smart eviction without routing still sees excessive swap rates.

Hot Path Optimization

For adapters that consistently receive high traffic, operators can configure permanent residency — marking certain adapters as "pinned" so they are never evicted from GPU memory. This is particularly useful for the top 2–3 adapters that may account for 60–80% of total traffic. Pinned adapters are pre-loaded at pod startup and excluded from the eviction pool, ensuring that the majority of requests always find their adapter in the hot path.

The EPP is aware of adapter pinning. When scoring candidates, it knows that a pinned adapter on a pod has zero risk of eviction, which contributes to a higher effective affinity score compared to a non-pinned adapter that might be evicted before the request reaches the pod.

05 Production Patterns

Cache-aware LoRA routing is not a set-and-forget feature. To extract maximum performance in production, operators need strategies for adapter lifecycle management, monitoring, and graceful degradation. The following patterns have emerged from real-world deployments.

Adapter Preloading Strategies

Rather than waiting for the first request to trigger an adapter load (cold start), production deployments use preloading to ensure adapters are already in GPU memory before traffic arrives. Three preloading strategies are common:

  • Time-of-day prediction: Many adapter traffic patterns are highly predictable. Customer support adapters see peak usage during business hours. Code assistant adapters spike during developer work hours (9 AM–6 PM local time). A preloading controller can use historical request logs to predict which adapters will be needed in the next time window and load them proactively at the start of each shift.
  • Warm-on-registration: When a new adapter is registered in the system (for example, after a fine-tuning run completes), immediately load it onto one or more pods. This eliminates cold-start latency for the first user who tests the new adapter.
  • Follow-the-sun: For global deployments, preload regional adapters based on the active time zone. As the workday ends in Europe, EU-specific adapters are deprioritized in favor of APAC adapters that will see growing traffic.

Hot/Cold Adapter Tiering

Not all adapters need to be in GPU memory at all times. A tiered storage strategy minimizes memory pressure while keeping swap latency manageable:

  • Hot tier (GPU HBM): Adapters currently loaded in GPU memory and actively serving requests. These serve with zero swap latency. Capacity: 4–64 adapters per pod, depending on rank and memory budget.
  • Warm tier (CPU host memory): Adapters that are not currently in GPU memory but are stored in the pod's host RAM (typically 256 GB–1 TB). Loading from CPU to GPU takes 3–20 ms over PCIe Gen5. Capacity: hundreds to thousands of adapters per node.
  • Cold tier (NVMe/network storage): Adapters stored on the node's NVMe SSD or on a shared storage system (like an S3-compatible object store or a distributed filesystem). Loading from NVMe takes 10–50 ms; from network storage, 50–200 ms. Capacity: effectively unlimited.

The tiering system is transparent to the EPP. The pod's state report includes the tier of each adapter, and the EPP's affinity scoring accounts for the expected swap latency at each tier (reflected in the graduated affinity scores from 1.0 for hot to 0.4 for warm to 0.0 for cold).

Adapter Affinity Groups

Some adapters are frequently requested in sequence. For example, a document processing pipeline might first invoke a classification-v2 adapter to categorize a document, then invoke a extraction-legal-v3 or extraction-medical-v1 adapter based on the classification result. If these adapters are on different pods, the pipeline incurs inter-pod latency and potentially triggers swaps on the second pod.

Affinity groups address this by hinting to the EPP that certain adapters should be co-located. The EPP factors group membership into its scoring: if a request for extraction-legal-v3 arrives and Pod A has both the adapter and its group peer classification-v2 loaded, Pod A gets a small bonus over Pod B that only has extraction-legal-v3. Over time, this causes pipeline-related adapters to naturally co-locate.

Graceful Degradation

When all pods are at adapter capacity and no pod has the requested adapter loaded, the system must degrade gracefully rather than failing. llm-d implements a multi-level degradation strategy:

  1. Evict and load: Route to the pod with the lowest-value adapter to evict (LRU or frequency-based). The request pays the swap penalty but succeeds.
  2. Queue with timeout: If all pods are actively swapping, queue the request with a configurable timeout (default: 5 seconds). This avoids cascading swaps.
  3. Fallback to base model: If the timeout expires and no pod can serve the adapter, fall back to serving the request with the base model (no adapter). The response quality may be slightly lower, but the user gets a response rather than an error.
  4. Signal scaling: Emit a Kubernetes custom metric indicating adapter pressure. Horizontal pod autoscalers can use this signal to scale the pod pool, adding capacity for the growing adapter set.

Monitoring: Key Metrics

Effective adapter-aware routing requires monitoring metrics that traditional serving setups do not track. The critical metrics to watch:

  • Swap rate (swaps per second per pod): The primary indicator of routing effectiveness. Healthy systems show swap rates below 0.1 swaps/sec/pod (fewer than 1 swap every 10 seconds). Rates above 1.0 indicate routing problems or insufficient adapter capacity.
  • Adapter residency time: How long each adapter stays loaded before being evicted. Short residency times (under 30 seconds) suggest thrashing — either too many adapters for the available slots, or poor routing affinity.
  • Cache hit ratio: The percentage of requests that find their adapter already loaded on the routed pod. Targets of 90%+ are achievable with EPP routing; below 80% suggests configuration tuning is needed.
  • Swap queue depth: The number of requests waiting for an adapter swap to complete. Consistently non-zero values indicate that swap latency is becoming a bottleneck.
  • Adapter popularity distribution: The request frequency distribution across adapters. Highly skewed distributions (a few adapters getting 80% of traffic) are easy for the routing layer to optimize. Uniform distributions (all adapters equally popular) are harder and may require more pods.

06 Comparison with Other Approaches

llm-d's EPP is not the only system that addresses multi-LoRA serving challenges. Several research projects and production systems have tackled different aspects of the problem. Understanding where each approach fits helps clarify llm-d's distinctive contribution.

Punica: SGMV Kernels for Batched LoRA

Punica introduced Segmented Gather Matrix-Vector (SGMV) multiplication — custom CUDA kernels that efficiently batch LoRA computations across multiple adapters in a single GPU kernel launch. Instead of running separate kernel calls for each adapter in a multi-adapter batch, SGMV gathers the appropriate adapter weights for each sequence, computes the low-rank products, and scatters the results back, all in one fused operation. This reduces kernel launch overhead and improves GPU utilization when serving heterogeneous adapter requests.

However, Punica focuses exclusively on the compute efficiency of multi-adapter batching. It does not address the routing question — which pod should receive which request. A cluster running Punica-optimized serving engines still needs an intelligent routing layer to avoid unnecessary adapter swaps between pods.

S-LoRA: Unified Paging for Adapter Weights

S-LoRA takes a memory management approach, applying the same paging concepts that vLLM uses for KV cache to adapter weights themselves. Adapters are stored in a shared, paged memory pool rather than in fixed-size slots. This allows more efficient memory utilization: a rank-64 adapter does not waste the same amount of memory as allocating a fixed slot sized for the largest possible adapter. S-LoRA also introduces adapter weight parallelism, distributing adapter computations across tensor-parallel groups.

Like Punica, S-LoRA's contributions are at the per-node serving level. It makes individual pods better at hosting many adapters simultaneously, but it does not provide inter-pod routing intelligence.

dLoRA: Dynamic LoRA Loading

dLoRA addresses dynamic adapter lifecycle management — the process of loading, unloading, and migrating adapters across GPUs in response to changing traffic patterns. dLoRA introduces a controller that monitors adapter request rates and proactively migrates adapters between GPUs to minimize swap latency. It also supports adapter replication, placing copies of popular adapters on multiple GPUs for load distribution.

dLoRA's approach is the closest to llm-d's EPP in spirit, as it considers the routing/placement problem rather than just the serving problem. The key difference is scope: dLoRA operates within a single multi-GPU node, while the EPP operates at the Kubernetes cluster level across potentially hundreds of pods. dLoRA also lacks the multi-signal scoring that the EPP uses — it routes based primarily on adapter presence, without considering KV cache affinity, queue depth, or GPU utilization as weighted signals.

How llm-d's EPP Fits In

The critical insight is that the EPP is complementary to these serving-level optimizations, not competitive with them. The EPP works at the routing layer, deciding which pod receives each request. The serving engines on those pods can incorporate Punica-style SGMV kernels for efficient multi-adapter batching, S-LoRA-style paged memory management for maximizing adapter capacity, and dLoRA-style intra-node adapter migration for multi-GPU optimization — all simultaneously.

In practice, llm-d's integration with vLLM means that many of these optimizations are already available through vLLM's multi-LoRA implementation, which has incorporated ideas from both Punica and S-LoRA. The EPP adds the cluster-level routing intelligence that none of these systems provide on their own.

07 Real-World Case Studies

The following case studies illustrate how cache-aware LoRA routing transforms production workloads. Each demonstrates a different traffic pattern and shows the quantitative impact of adapter-affinity routing compared to naive load balancing.

Customer Support Platform

A large B2B SaaS platform provides AI-powered customer support for 50+ enterprise customers. Each customer has a custom LoRA adapter trained on their product documentation, support ticket history, and brand voice guidelines. The platform serves approximately 2,000 requests per minute across a cluster of 12 vLLM pods running a 70B base model with INT4 quantization.

Before EPP: Round-Robin Routing

With standard Kubernetes load balancing, each pod received a uniform mix of requests across all 50+ adapters. Each pod could hold approximately 8 adapters in GPU memory at once. The math was straightforward: 50 adapters across 8 slots meant that roughly 84% of requests required an adapter swap. During peak hours (when all 50 adapters were active), the effective swap rate exceeded 1,600 swaps per minute cluster-wide. The convoy effect was severe — during swap cascades, P95 latency spiked from 400 ms to over 2 seconds, and several customer SLAs were at risk.

After EPP: Adapter-Affinity Routing

After deploying llm-d's EPP with adapter-affinity scoring, the adapter traffic naturally clustered. The top 10 customers accounted for 70% of traffic, and their adapters became pinned residents on 4 dedicated pods. The remaining 40+ lower-traffic adapters distributed across the other 8 pods, with the LRU eviction policy handling the long tail.

Code Assistant Suite

A developer tools company offers AI code assistance with 4 specialized adapters: Python, JavaScript, Rust, and Go. Each adapter is fine-tuned on language-specific codebases, documentation, and idioms. The service runs on 6 pods with an H100 GPU each, serving a mix of IDE plugin users and CI/CD pipeline integrations.

Traffic follows developer work patterns with strong temporal locality. During a typical workday, a developer works primarily in one language for extended periods, producing bursts of requests for a single adapter. But different developers work in different languages, so the aggregate traffic across the cluster is mixed.

Without affinity routing, each pod received a mix of all 4 languages. Despite there being only 4 adapters (well within each pod's capacity), the random distribution meant that pods were constantly activating and deactivating adapters in their batch pipeline, paying the warmup penalty repeatedly. With EPP routing, Python and JavaScript traffic (the two highest-volume adapters) were each given natural affinity to 2 pods, while Rust and Go shared the remaining 2 pods. The swap rate dropped from 45% to 7%, and the warmup overhead that was adding 50–100 ms to post-swap requests was nearly eliminated. P99 latency improved from 1.1 seconds to 320 ms.

Domain-Specific RAG: Legal Research Platform

A legal technology company provides a research assistant that combines retrieval-augmented generation with jurisdiction-specific fine-tuning. The platform hosts 12 jurisdiction-specific adapters (Federal, California, New York, Texas, Delaware, Illinois, plus 6 more). Each adapter is trained on the case law, statutes, and legal conventions of its jurisdiction, so that generated legal analyses use the correct citation formats, precedent hierarchies, and procedural terminology.

Legal research has strong session locality: a lawyer working on a California case makes dozens of queries in sequence, all targeting the ca-law-v4 adapter. But the firm has lawyers across all 12 jurisdictions, creating cross-adapter traffic at the cluster level.

The deployment uses EPP with adapter affinity routing plus time-of-day preloading. Because the firm's offices span Eastern, Central, and Pacific time zones, the preloading controller loads jurisdiction adapters based on which offices are currently active. When the New York office opens at 9 AM ET, the ny-law-v3 and federal-law-v5 adapters are preloaded onto pods closest to the eastern availability zone. The result: cold-start latency for the first query of the day dropped from 1.2 seconds (adapter load from NVMe + warmup) to 180 ms (adapter already hot in GPU memory, only KV cache computation needed). Session-continuation latency for subsequent queries in the same jurisdiction is consistently under 200 ms, because the EPP routes every request in the session to the same pod.

Pattern: Session Affinity Amplifies Adapter Affinity

When adapter traffic has session-level locality (the same user repeatedly requesting the same adapter), combining adapter affinity with session stickiness in the EPP produces compounding benefits. The EPP routes a session's first request based on adapter affinity, and subsequent requests in the session inherit both adapter and KV cache affinity. This is the ideal case for multi-turn conversations and iterative research workflows.

08 Disaggregated Prefill/Decode with LoRA

llm-d's architecture supports prefill/decode (P/D) disaggregation, a technique that separates the two phases of LLM inference onto different GPU pools optimized for each phase's resource profile. When combined with adapter-aware routing, P/D disaggregation introduces unique challenges and opportunities that merit dedicated discussion.

Why Disaggregate?

LLM inference has two computationally distinct phases:

  • Prefill processes the entire input prompt in parallel. It is compute-bound — high FLOPS utilization, high arithmetic intensity, and it benefits from GPUs with maximum compute throughput. Prefill runs once per request. For a 4,096-token prompt on a 70B model, prefill typically takes 500–1,500 ms.
  • Decode generates output tokens autoregressively, one at a time. It is memory-bandwidth-bound — each token generation reads the full model weights but performs relatively little computation per weight. Decode runs once per output token and benefits from GPUs with high memory bandwidth and efficient batching across many concurrent sequences.

When both phases run on the same GPU, they compete for resources inefficiently. Prefill's compute-heavy workload underutilizes memory bandwidth, while decode's bandwidth-heavy workload underutilizes compute. Disaggregation assigns each phase to GPU pools configured for its workload, connected by a KV cache transfer mechanism that ships the computed key-value tensors from the prefill pod to the decode pod.

Both Pods Need the Correct Adapter

Disaggregation complicates adapter routing because both the prefill and decode pods must have the correct LoRA adapter loaded. The reason traces back to Section 1's discussion of how adapters modify attention projections:

  • The prefill pod needs the adapter to correctly compute the KV cache entries. Because LoRA modifies the Q, K, and V projection matrices, the resulting key and value tensors are adapter-specific. A KV cache computed with adapter A cannot be correctly consumed by a decode pod running adapter B.
  • The decode pod needs the same adapter for two reasons: (1) to generate tokens that are consistent with the adapted model's behavior, and (2) to correctly compute each new KV cache entry during autoregressive generation, extending the cache produced by prefill.

If the routing layer sends a request to a prefill pod with the right adapter but a decode pod without it, the decode pod must perform an adapter swap before it can begin generating tokens. This swap happens at the worst possible time — the user is already waiting, the prefill cost has already been paid, and the KV cache is sitting in the decode pod's memory consuming space without being used.

Separate Affinity Maps for Prefill and Decode Pools

The EPP handles P/D disaggregation by maintaining two independent adapter affinity maps — one for the prefill pool and one for the decode pool. Each map tracks which adapters are loaded on which pods within its respective pool. When a request arrives:

  1. Prefill routing: The EPP scores prefill pods using the standard composite formula, selecting the prefill pod with the highest adapter affinity and best load balance.
  2. Decode pod pre-selection: Before dispatching to prefill, the EPP identifies a decode pod with the same adapter loaded. This pre-selection ensures that the KV cache handoff will not be blocked by an adapter swap on the decode side.
  3. KV cache transfer: After prefill completes, the computed KV cache is transferred to the pre-selected decode pod via RDMA, NVLink, or TCP, depending on the network topology.
  4. Decode execution: The decode pod receives the KV cache and begins token generation immediately, with zero adapter swap overhead.

The pre-selection step is key. Without it, the system would need to find a decode pod with the right adapter after prefill completes, which could fail under load and force a swap at the worst time. By pre-selecting the decode pod at routing time, the EPP guarantees a swap-free end-to-end path.

The Compounding Benefit

P/D disaggregation and adapter-aware routing produce compounding benefits when used together. Disaggregation alone can improve throughput by 30–60% for mixed workloads by allowing each GPU pool to be configured optimally for its workload phase. Adapter-aware routing alone can reduce swap rates from 70%+ to under 10%, recovering throughput lost to swap overhead. Together, they enable a system where:

  • Prefill pods can be more aggressively specialized by adapter, because prefill is a short, bursty operation. A prefill pod that handles only 3–4 adapters completes each prefill quickly and moves to the next request.
  • Decode pods can hold more adapters simultaneously, because decode's lower memory pressure (smaller per-request working set) leaves more HBM available for adapter weight storage.
  • The effective adapter capacity of the cluster increases because the two pools can be sized independently, and the adapter working sets can overlap differently across the two pools based on traffic patterns.
Performance Impact

In benchmarks with 20 active adapters across an 8-pod cluster (4 prefill, 4 decode), the combination of P/D disaggregation with EPP adapter-affinity routing achieved 2.3x higher throughput and 65% lower P99 latency compared to colocated serving with round-robin routing. Neither optimization alone achieved more than 1.5x throughput improvement, demonstrating the compounding effect.
