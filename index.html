The Problem
Adapter swaps cost more than the swap itself
In multi-tenant LLM serving, different users need different LoRA adapters. Loading a new adapter takes 500-2000ms -- and it drains the entire batch pipeline, blocking every other request on that pod.
❌ Round-Robin Routing

Picks the next pod in rotation regardless of which adapters are loaded. Completely ignores cache state, GPU load, and adapter affinity.

Scores every candidate pod using adapter affinity, KV cache warmth, queue depth, and GPU health -- then picks the optimal target.

The Scoring Engine
How the EPP scores every pod
The Endpoint Picker computes a weighted composite score for each candidate pod, then routes to the winner.
Adapter Affinity
50%
Does this pod already have the requested adapter loaded? Avoids swap overhead entirely.
KV Cache Match
20%
Does this pod have cached key-value pairs from prior requests? Reduces time-to-first-token.
Load / Queue Depth
15%
How busy is this pod? Avoids overloading pods that are already saturated.
GPU Health
15%
Current resource availability. Healthier pods with more headroom are preferred.
composite = adapter_affinity(0.50) + cache_match(0.20) + 1/load(0.15) + gpu_health(0.15)

War Story: The 3 AM Swap Storm
A batch job kicks off 200 requests for a rare adapter. Round-robin spreads them across all 8 pods. Every pod starts swapping simultaneously. Getting Started
Start routing smarter today
llm-d is a CNCF Sandbox project. It runs on Kubernetes and integrates with vLLM, HuggingFace, and any OpenAI-compatible API.
  1. Deploy llm-d on your Kubernetes cluster with Helm or Kustomize
  2. Register your LoRA adapters -- point to HuggingFace or local storage
  3. Configure the EPP -- set adapter affinity weights, pod count, and GPU type
  4. Send requests -- the EPP handles routing automatically, scoring every pod per-request
  5. Monitor -- built-in Prometheus metrics for swap rate, cache hit rate, and P95 latency
