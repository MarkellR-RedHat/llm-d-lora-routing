How llm-d Cache-Aware LoRA Routing Works

The Problem: In multi-tenant LLM serving, different users need different LoRA adapters (fine-tuned model layers). Loading a new adapter onto a GPU pod takes 500-2000ms and blocks the pod during the swap. Naive round-robin routing ignores which adapters are already loaded, causing excessive and unnecessary swaps.

The Solution: llm-d's Endpoint Picker (EPP) scores each candidate pod using multiple signals before routing:

Adapter Affinity (50%) — Does this pod already have the requested adapter loaded? This is the highest-weight signal. Routing to a pod that already has the adapter avoids swap overhead entirely.

KV Cache Match (20%) — Does this pod have relevant cached key-value pairs from prior requests with this adapter? Reusing KV cache reduces time-to-first-token for similar prompts.

Load / Queue Depth (15%) — How busy is this pod? The EPP avoids overloading pods that are already saturated, balancing throughput across the cluster.

Health / GPU Utilization (15%) — What is the pod's current resource availability? Healthier pods with more headroom are preferred.

Composite Score — The EPP combines these weighted signals into a single score, picking the pod that minimizes total latency while maximizing resource reuse.

This visualizer runs the same request stream through both strategies simultaneously, so you can see the difference in adapter swaps, latency, and throughput in real time. Watch how the smart routing concentrates each adapter type on fewer pods, while round-robin scatters them everywhere.
